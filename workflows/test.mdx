The `/test` command generates tests, runs existing tests, or checks test coverage. It applies the [testing-patterns skill](/skills/testing-patterns) to create comprehensive test suites.

## Sub-commands

```bash
/test                # Run all tests
/test [file/feature] # Generate tests for specific target
/test coverage       # Show test coverage report
/test watch          # Run tests in watch mode
```

## How it works

### Generate tests

When you test a file or feature:

<Steps>
  <Step title="Analyze the code">
    - Identify functions and methods
    - Find edge cases
    - Detect dependencies to mock
  </Step>
  
  <Step title="Generate test cases">
    - Happy path tests
    - Error cases
    - Edge cases
    - Integration tests (if needed)
  </Step>
  
  <Step title="Write tests">
    - Use project's test framework (Jest, Vitest, etc.)
    - Follow existing test patterns
    - Mock external dependencies
  </Step>
</Steps>

## Test patterns

All generated tests follow the **AAA pattern** (Arrange-Act-Assert):

```typescript
describe('AuthService', () => {
  describe('login', () => {
    it('should return token for valid credentials', async () => {
      // Arrange
      const credentials = { 
        email: 'test@test.com', 
        password: 'pass123' 
      };
      
      // Act
      const result = await authService.login(credentials);
      
      // Assert
      expect(result.token).toBeDefined();
    });

    it('should throw for invalid password', async () => {
      // Arrange
      const credentials = { 
        email: 'test@test.com', 
        password: 'wrong' 
      };
      
      // Act & Assert
      await expect(
        authService.login(credentials)
      ).rejects.toThrow('Invalid credentials');
    });
  });
});
```

## Key principles

<Note>
  **Test behavior not implementation**: Focus on what the code does, not how it does it.
</Note>

<Note>
  **One assertion per test**: When practical, each test should verify one specific behavior.
</Note>

<Note>
  **Descriptive test names**: Test names should clearly state what is being tested.
</Note>

<Note>
  **Mock external dependencies**: Isolate the code under test by mocking APIs, databases, etc.
</Note>
